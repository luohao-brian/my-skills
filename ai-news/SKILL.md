# AI 新闻简报

产出中文版 AI 新闻简报。核心不是“搜到一些链接”，而是按稳定 workflow 获取一手来源、用搜索补漏和交叉验证，再按统一格式成稿。

开始执行前，必须先读：

- [sources.md](sources.md)：来源类型、品牌入口、搜索引擎分工
- [format.md](format.md)：最终输出格式

硬规则：

1. 没有先读 `sources.md`，不要开始抓取。
2. 没有先读 `format.md`，不要开始写结果。
3. 开始写最终 markdown 前，必须再读一遍 [format.md](format.md)。
4. 如果入口失败、跳转、返回 landing page、返回壳页面或信息价值存疑，先回到 [sources.md](sources.md) 查对应入口和其它入口，不要自己临时决定。
5. 用户说“今天 / 今日 / 最新 / 刚刚 / 日报”时，最终收录内容必须严格落在目标时间窗内；超出时间窗的条目默认删除，不得用旧闻补齐版面。
6. 如果目标时间窗内有效内容不足，直接说明“当前时间窗内公开更新较少”，不要偷偷放宽到更早日期。
7. 只读取列表页不够。进入最终简报的每一条新闻，都必须至少有一个带明确发布时间的详情页或官方发布页作为依据。

## 基本原则

1. 默认使用最通用的 `web_search` 和 `web_fetch` 做编排，不要假设存在额外专用工具。
2. 同时扫描官网、权威新闻网站、权威论坛、新闻聚合网站和开源模型 / 开源项目入口，再用搜索补漏和交叉验证。
3. 成稿时使用信息最完整、发布时间最明确、最接近原始发布的链接。
4. 最终结果必须是中文 markdown 简报，按分类组织，不能输出线性列表。

## 目标

生成一份 markdown 简报，满足：

1. 必须按分类组织，不允许只输出线性列表。
2. 每条新闻必须包含标题、摘要、发布时间、来源链接。
3. 标题和摘要用中文。
4. 同一条新闻只能出现一次，不要跨分类重复。
5. 使用信息最完整、发布时间最明确、最接近原始发布的链接。

## 默认时间窗口

如果用户没有指定日期，默认整理最近 `24-48 小时` 内仍有参考价值的 AI 新闻。

时间判断规则：

- 默认先看最近 48 小时
- 如果 48 小时内信息太少，再明确说明覆盖不足，不要偷偷放宽到更久
- 如果用户指定“今日 / 昨日 / 本周”，按用户要求执行，并在标题里写绝对日期
- 发布时间展示默认用相对时间，例如 `3 小时前`；如果有绝对时间，写成 `3 小时前（2026-03-29 14:30）`
- “今天 / 今日 / 最新 / 刚刚”默认理解为：先覆盖当天自然日，再允许补充最近 `24 小时` 内仍在今天话题窗口中的更新；默认不要收录 3 天前、1 周前、1 月前的旧闻。

## 分类

- 大模型
- Agent
- 融资 / 商业
- 安全 / 治理
- 应用 / 产品
- 开源

如果某个分类在目标时间窗内没有足够有效内容，可以省略；但必须先完成全部来源类型扫描和搜索补漏，不能先入为主地判定“没有新闻”。

## 工作流

### 0. 今天 / 最新场景先做结构化搜索

用户说“今天 / 今日 / 最新 / 刚刚 / 日报”时，先做结构化搜索建立候选集：

- `tavily`：`intent=news`，并显式给 `freshness=pd` 或 `date_after/date_before`
- `bocha`：同样显式给时间范围
- 国际英文动态再补 `brave`

先确认“今天窗口内有哪些事件”，再回到固定来源和详情页补齐信息。

### 1. 扫全部固定来源

先读 [sources.md](sources.md) 里的全部来源类型。产品和技术相关内容重点覆盖这些品牌入口：

- OpenAI
- Anthropic
- Google AI / Google DeepMind
- xAI
- MiniMax
- 智谱 AI
- Kimi / Moonshot
- 阿里 AI
- 腾讯 AI
- 火山引擎

执行步骤：

1. 扫官网 / 官方发布入口。
2. 扫权威新闻网站。
3. 扫权威论坛 / 社区。
4. 扫新闻聚合网站。
5. 扫开源模型 / 开源项目入口。
6. 遇到官方站点返回 challenge、壳页面、列表字段不足或详情页打不开时，立即改用官方域名、品牌名、产品名定向搜索回源。
7. 遇到 MiniMax、Moonshot、火山引擎这类单个入口更新不连续或抓取结果不完整的品牌，同一品牌至少再补一轮官方域名定向搜索。

### 2. 对候选做交叉验证

对进入候选集的条目，补齐这三件事：

- 确认发布时间
- 确认事件是否真实发生
- 确认最终成稿链接是否完整、可引用

### 3. 用搜索做补漏和交叉验证

固定来源扫描之后，继续用搜索补漏，不停留在已有入口。

开始搜索前，先回看 [sources.md](sources.md) 里的“搜索引擎分工”和“最低搜索覆盖要求”，不要临时发明搜索策略。

搜索引擎使用要求：

- `tavily`：做宽召回、时效新闻和深度补漏
- `bocha`：做中文宽度补充和中文站点补漏
- `brave`：做英文国际动态、英文交叉验证和官方账号补充

最低要求：

- 至少覆盖 `tavily`、`bocha`
- 对英文国际动态或需要英文交叉验证的分类，再补 `brave`
- 每个重要分类至少做一轮中文 query 和一轮英文 query
- 对重点公司至少做一轮“官方域名 + 产品名”定向检索
- 用户说“今天 / 今日 / 最新”时，`tavily` 和 `bocha` 的查询必须显式带 `freshness=pd` 或等价的 `date_after/date_before`；不要只把“今天”写进 query。

搜索顺序：

1. 分类主题宽召回
2. 品牌 / 产品 / 站点定向检索
3. 日期 / 语言 / 站点定向检索

不要只搜 `AI news` 这种宽泛 query。要拆成：

- 公司名 + 产品名
- 分类主题词
- 日期词面 query

示例：

- `site:openai.com GPT-5.4 OR ChatGPT 2026-03-29`
- `2026-03-29 AI agent news`
- `2026年3月29日 大模型 发布`

### 4. 抓详情页并过滤

候选新闻进入成稿前，要尽量补齐：

- 中文标题
- 中文摘要
- 发布时间
- 最终来源链接

过滤要求：

- 只保留 AI 相关内容
- 同源重复报道只保留信息最完整的一条
- 二手转载让位于原始来源或一手报道
- 无发布时间、无正文支撑、明显营销软文不要收录
- 对只有单点来源的信息，至少再做一轮搜索交叉验证
- 不在目标时间窗内的条目不要收录
- 仅有列表页摘要、没有详情页正文支撑的条目不要收录

### 5. 成稿前重新读 format.md

开始写最终答案前，必须再次读取 [format.md](format.md)，不要只凭记忆输出。

最低要求：

- 分类是二级标题
- 每条新闻是三级标题
- 每条固定三行字段：`摘要`、`发布时间`、`来源链接`
- 不输出抓取日志、命令回显、过程噪音
- 对“今天 / 今日 / 最新”场景，输出前逐条核对发布时间；任何超窗条目都必须删掉

## 执行清单

执行时按这个顺序自检：

1. 官网、权威新闻网站、权威论坛、新闻聚合网站是否都已扫描
2. 是否至少使用了 `tavily`、`bocha`
3. 英文国际动态是否补了 `brave`
4. 详情页字段是否补齐
5. 最终输出前是否重新读取了 [format.md](format.md)
6. “今天 / 最新”场景下，是否每条都在目标时间窗内
7. 是否避免了仅凭列表页摘要直接成稿
